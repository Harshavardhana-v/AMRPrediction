AMR-Predict v2.0 — Product Requirements Document

1. Executive Summary
AMR-Predict v1.0 established a working proof-of-concept: a React + Flask + XGBoost pipeline that accepts six clinical inputs and returns a binary Resistant / Susceptible prediction. While the architecture is sound, the current product has significant gaps that prevent real-world clinical adoption — most critically, the model is trained on fully synthetic data, offers no explanation for its decisions, and supports only a single binary outcome.
This PRD defines Version 2.0 — a three-phase enhancement roadmap organized around three pillars:
Pillar 1 — Clinical Credibility: Real data retraining, SHAP explainability, and multi-drug resistance profiling.
Pillar 2 — Operational Robustness: Prediction history, input validation, rate limiting, Docker deployment, and environment config.
Pillar 3 — UX Excellence: Feature importance visualization, patient summary cards, PDF export, history dashboard, and improved form UX.

2. Problem Statement
2.1 Clinical Gap
Standard culture-based susceptibility testing takes 24–72 hours. AMR-Predict v1.0 demonstrated that ML can reduce this to under 2 seconds, but it cannot yet be trusted in a clinical setting because:

The model was trained on randomly generated synthetic data with no basis in real patient outcomes.
The prediction is a black box — clinicians cannot see which clinical factors drove the decision.
Only a single binary outcome is returned; real treatment requires resistance profiles across multiple antibiotics.
There is no audit trail, making the tool incompatible with clinical governance requirements.

2.2 Technical Debt

The API URL is hardcoded as http://127.0.0.1:5000 in the frontend, breaking all non-local deployments.
There is no server-side input validation — malformed requests can crash the Flask service.
No containerization means the setup process requires manual environment management.
The /predict endpoint has no rate limiting, making it vulnerable to abuse.


3. Goals & Non-Goals
3.1 Goals

Real Data — Retrain the XGBoost model on the real E. coli and Klebsiella datasets already present in the project.
Explainability — Add SHAP-based explainability to every prediction, surfaced in both the API response and as a visual chart in the frontend.
Multi-Drug — Expand from one binary outcome to a per-antibiotic resistance profile covering at least 5 common antibiotics.
Audit Trail — Persist every prediction to SQLite and expose a history dashboard in the frontend.
Production Hardening — Add input validation, rate limiting, environment variable configuration, and Docker support.
UX Improvements — Add a patient summary card, PDF report export, and interactive form improvements (sliders, tooltips).
Transparency — Add an About / Research page with model performance metrics and dataset citations.

3.2 Non-Goals (Out of Scope for v2.0)

Direct EHR / HL7 FHIR integration — v3.0 item.
Real-time genomic sequencing data input.
User authentication and multi-tenant patient record management.
Mobile native app (iOS/Android).
Any form of PII storage — all predictions are stored anonymously.

6. Detailed Functional Requirements
6.1 F-01 — Real Dataset Retraining

Load and merge ecoli_resistant_cleaned.csv and klebsiella_resistance_cleaned.csv from the dataset/ folder.
Perform EDA: report class distribution, missing value counts, and feature correlation matrix to console.
Apply SMOTE oversampling if class imbalance ratio exceeds 2:1.
Perform 5-fold stratified cross-validation. Log per-fold AUC and final mean ± std.
Save retrained model to models/amr_xgboost_model_v2.pkl with a companion models/model_meta.json (training date, dataset, metrics).
Add POST /retrain endpoint accepting { "api_key": "..." }. Return training metrics on success.

6.2 F-02 / F-03 — SHAP Explainability

Import shap.TreeExplainer. Instantiate once at app startup alongside model loading.
On every /predict call, compute SHAP values for the input row and include a shap_values object in the response: { feature_name: shap_value, ... }.
In the React Result page, render a horizontal bar chart (recharts BarChart) where each bar represents one feature's SHAP value. Positive bars (red) push toward Resistant; negative bars (green) push toward Susceptible.
Display the feature's actual input value alongside each bar as a label.
Add a one-sentence plain-English interpretation above the chart (e.g., "Prior antibiotic use was the strongest driver of this prediction").

6.3 F-08 — Multi-Drug Resistance Profile

Train 5 independent XGBoost classifiers, one per antibiotic: Ciprofloxacin, Amoxicillin, Meropenem, Gentamicin, Ceftriaxone.
Store each model as models/amr_{drug}_{bacteria}_model.pkl.
The /predict endpoint returns a resistance_profile map: { "Ciprofloxacin": { "prediction": "Resistant", "probability": 0.87 }, ... }.
The primary prediction field remains for backward compatibility.
The frontend Result page renders the multi-drug grid as the primary output, with the binary verdict shown as a summary badge.

6.4 F-10 / F-11 — Prediction History

Use SQLite via Python's built-in sqlite3 module. Create predictions.db on first run.
Schema: prediction_id (UUID), created_at (ISO timestamp), inputs (JSON blob), prediction, probability, risk_tier, shap_values (JSON blob).
GET /history returns paginated results (default 20 per page) with optional filter by infection_type and date range.
GET /history/:id returns the full record including SHAP values for re-rendering the explanation chart.
The frontend /history page renders a sortable table with columns: Date, Infection Type, Prediction, Risk Tier, Confidence. Clicking a row navigates to a read-only Result view for that prediction.

6.5 F-15 — PDF Report Export

Use jsPDF (client-side) combined with html2canvas to capture the rendered Result page.
The PDF includes: hospital/app header, patient input summary table, prediction verdict with risk tier badge, SHAP feature chart image, confidence score, and timestamp.
Filename: amr_report_{timestamp}.pdf.
Export must complete in under 3 seconds on a modern browser.

13. Open Questions

Data Audit — Which 5 antibiotics should the multi-drug model cover? Depends on which resistance labels are present in the real dataset — a data audit is needed before Phase 2 begins.
Retrain API — Should /retrain accept a new dataset file upload (multipart), or only retrain on the static dataset/ folder? Recommend file upload for flexibility.
Deployment Mode — Is there a requirement to support concurrent users in a hospital LAN deployment? If yes, Gunicorn + PostgreSQL should replace Flask dev server + SQLite from Phase 1.
History Scoping — Should prediction history be user-scoped? Current design stores all predictions globally. If multi-user is needed, a simple session ID approach (no login) could be added in Phase 2.
